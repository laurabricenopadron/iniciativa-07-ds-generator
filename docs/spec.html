Generador de Design Systems con IA

A partir de la paleta y tipografía de una marca, la herramienta genera automáticamente un sistema de diseño completo — con reglas fijas de contraste y derivación de color — que se materializa en componentes reales dentro de Figma mediante un template publicado en la Community.

Programa: AI — DxD Empresa: Baufest Estado: En construcción avanzada Actualizado: 21 jul 2026

01Problem Statement — versión corta

Crear un design system inicial (paleta, tipografía, componentes) le toma hoy a un diseñador entre 40 y 50 horas de trabajo manual: investigar referencias, probar paletas, definir cada componente por separado. Existen herramientas de IA genéricas (ChatGPT, Gemini, Google Stitch) que aceleran partes sueltas del proceso, pero ninguna resuelve el problema completo — no aplican reglas de contraste de forma sistemática ni adaptan todos los componentes bajo el mismo criterio, así que el resultado sigue siendo parcial y requiere retrabajo manual.

Esta iniciativa está dirigida a diseñadores UX/UI y equipos pequeños que necesitan un punto de partida real, no inspiración suelta. A partir de los colores y tipografía que ya tiene una marca (o de una breve descripción si es nueva), la herramienta genera automáticamente un sistema de diseño completo — con reglas fijas de contraste y derivación de color aplicadas sobre el 100% de los componentes — listo para exportar a Figma.

02Problem Statement — versión extendida

Definir un design system inicial (paleta de colores, tipografía y componentes base) es una de las tareas más lentas y repetitivas del proceso de diseño. No es un problema anecdótico: estudios de la industria sitúan la construcción de un design system mediano en 40-50 horas de trabajo, y casos documentados de equipos reales (Lucid Software, Lago) hablan de semanas de dedicación de diseñadores y desarrolladores antes de poder empezar a diseñar producto real.

El problema no es solo de tiempo, sino de consistencia: cuando el diseñador define colores, tipografía y componentes a mano, es fácil que las decisiones no queden documentadas de forma reutilizable (qué color va dónde, qué contraste cumple accesibilidad, qué radius corresponde a cada componente), lo que genera drift entre Figma y el código más adelante.

Qué pasa hoy

Hoy existen herramientas de IA genéricas que ya se usan para acelerar partes sueltas de este proceso — sugerir una paleta, tirar ideas de tipografía. Pero ninguna resuelve el problema completo: no aplican reglas de contraste de forma sistemática, no derivan los roles de color de manera consistente, ni adaptan todos los componentes base bajo el mismo criterio. El resultado sigue siendo parcial — sirve como inspiración, no como sistema.

Esta iniciativa propone una herramienta que, a partir de los atributos de marca que el usuario ya tiene (colores, tipografía) o de una breve descripción de personalidad de marca, genere automáticamente un sistema de diseño inicial completo y coherente — paleta con roles semánticos, escala tipográfica, y un set de componentes base (botón, input, card, badge) ya adaptados a esos tokens — listo para exportarse a Figma en un formato estándar (compatible con W3C Design Tokens). El objetivo no es reemplazar el criterio del diseñador, sino comprimir de días/semanas a minutos la fase de fundamentos.

Para quién

Diseñadores UX/UI y equipos pequeños que necesitan arrancar un proyecto de branding o producto sin partir de cero.

Validación de mercado

Este no es un problema teórico. Google ya invierte en la misma dirección con Stitch (Google Labs), que genera UI completa a partir de prompts usando Gemini. Sin embargo, reseñas independientes de Stitch señalan su principal debilidad: sin reglas de marca deterministas, los colores se desvían de los sistemas de marca y los componentes pierden consistencia. Esta iniciativa toma esa lección: acota el alcance a los fundamentos del design system y aplica reglas fijas de derivación de color y contraste.

03El producto: dos piezas que trabajan juntas

El entregable de esta iniciativa no es una sola aplicación, sino dos piezas que funcionan en conjunto. Esta es la evolución clave del proyecto respecto al planteo inicial, validada en la Revisión 1.3.

  • Pieza 1 — Design Gen (la app web): genera las decisiones de diseño con IA (paleta con roles, contraste, escala tipográfica, radius por componente) y las exporta como JSON de tokens en formato W3C DTCG.
  • Pieza 2 — Template de Figma Community: un archivo publicado con los 4 componentes (button, input, card, badge) ya construidos y con cada propiedad vinculada a variables de Figma. Al importar el JSON generado, las variables se actualizan y todos los componentes se re-tematizan solos con la marca.
Por qué dos piezas

La IA genera las decisiones (qué colores, qué contraste, qué radius); Figma las materializa en componentes nativos. Es el mismo patrón que usa Material Theme Builder de Google: se generan los tokens, y los componentes del template se adaptan. Así, los componentes generados por la IA terminan existiendo como componentes reales en Figma, adaptados a la marca — cumpliendo la meta de la iniciativa de forma usable y distribuible.

El template se publicó en Figma Community, de modo que cualquier diseñador puede duplicarlo con un click e importar su propio JSON. El JSON se importa con el plugin gratuito Design Tokens Manager, que lee el formato W3C DTCG sin necesidad de ajustes.

2.2 Reglas fijas del sistema

Estas reglas no se le preguntan al usuario, se imponen: van directo en el prompt a Gemini como restricciones, para que el resultado sea predecible y no varíe caóticamente entre generaciones.

  • Roles de color: todo color adicional (on-primary, primary-container, neutros, estados) se deriva del primario/secundario ingresado — nunca se inventa un color nuevo sin relación con la marca.
  • Contraste: todo par texto/fondo se calcula para cumplir WCAG AA (4.5:1) automáticamente.
  • Badge: siempre usa radius.full (pill), independientemente del preset elegido.
  • Colores semánticos (success / warning / error): son fijos por el sistema, no se derivan de la marca, para no comprometer su legibilidad universal.
  • Preset de forma → tokens de radius: el usuario no fija un número de px, elige una filosofía (Sharp/Soft/Rounded/Full) que mapea a una tabla fija de valores por componente.

2.3 Export a Figma — cómo se materializa

El JSON generado por la app se convierte en un design system real dentro de Figma a través del template de Community. El mecanismo:

  • Variables como puente: el template tiene una colección de variables de Figma (color, radius y tamaños tipográficos) cuyos nombres coinciden 1 a 1 con la estructura del JSON (color/primary, radius/button, typography/scale/body, etc.).
  • Componentes atados a variables: los 4 componentes no tienen valores fijos — cada fill, radius y tamaño de texto está vinculado a su variable. Ningún valor "pintado a mano".
  • Re-tematización automática: al importar el JSON con el plugin Design Tokens Manager (sobre la colección existente, sin borrarla), las variables se actualizan y los componentes cambian solos. Validado con múltiples marcas distintas.
  • Formato sin fricción: el plugin lee el W3C DTCG ($value, $type) y convierte las dimensiones string ("6px") en variables numéricas correctas — el JSON no requiere cambios.
  • Guía integrada en la app: la pantalla de resultados incluye una sección de instrucciones con los links directos al template y al plugin, para que el usuario complete el flujo sin documentación externa.

Esta elección de formato está validada por el mercado: Figma anunció soporte nativo de import/export de variables en formato W3C DTCG (en rollout hacia fin de 2026). El formato elegido es el que Figma mismo está adoptando como estándar.

2.5 Nota sobre datos

La capa gratuita de Gemini API puede usar los prompts y respuestas para mejorar los modelos de Google. Dado que esta demo trabaja con descripciones de marca de ejemplo (no datos reales de clientes), esto no representa un riesgo para el alcance del programa.

05Criterios de éxito

  1. ✅ Cumplido. El usuario puede generar un design system completo (paleta con roles, tipografía, y 4 componentes: botón, input, card, badge) en menos de 60 segundos desde que envía el formulario.
  2. ✅ Cumplido. El JSON de tokens exportado cumple la estructura W3C DTCG ($value, $type) y se importa en Figma sin errores — validado con el plugin Design Tokens Manager sobre el template de Community, re-tematizando los componentes con múltiples marcas.
  3. ✅ Cumplido. El 100% de las combinaciones texto/fondo generadas cumplen contraste WCAG AA (4.5:1), verificable de forma automática sin ajuste manual (función validateAndFixContrast).
  4. ✅ Cumplido. El usuario puede elegir entre 4 presets de forma (Sharp / Soft / Rounded / Full Round) y ver los 4 componentes actualizar su radius en el preview según la tabla fija definida.
  5. 🔜 En curso. El preview muestra el sistema generado correctamente en al menos 2 breakpoints (mobile y desktop), con los tokens de espaciado y tipografía adaptados a cada uno.
Alcance en implementación

Persistencia del historial de proyectos generados con PostgreSQL en Railway, integrando la base de datos al flujo para guardar y recuperar sistemas generados.
